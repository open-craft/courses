Tests

Test coverage is our insurance policy against upstream changes, and so we are careful to ensure that the features we write are tested well. You can find the commands to run the different types of tests in our documentation: Code quality and tests.

We care about things like this:

  • Test the required behaviour, not the way the code is written.
  • When implementing a feature for a client, ensure that the specific configuration settings that the client will rely on are well tested.

But we try not to:

  • Create brittle tests which will fail easily if someone changes anything related to the code.
  • Reproduce logic from the code in the tests. Hard-coded expected results in tests are far easier to debug than a function which reproduces the code functionality to generate the expected response.
